# Unified Draft Node Flow Design

Date: 2026-04-27

## Summary

Tritree should treat every new draft as a new tree node. AI option selection, user-edited drafts, diff-edited drafts, and user-written custom directions should all enter one shared creation flow:

1. Create the child node.
2. Move session state to that node.
3. Fill the node with a draft, either user-submitted or AI-generated.
4. Generate the next three options for that node.

This replaces the current split where direct draft editing mutates the active node while choice flows append nodes. The user mental model becomes simpler: every meaningful draft version is a node in the tree.

## Goals

- Add diff-view editing so users can revise a compared draft and save it as a child node.
- Make direct editing save as a child node instead of overwriting the current node.
- Preserve custom direction behavior while routing it through the same node lifecycle.
- Keep all newly generated draft nodes consistent in tree structure, session state, draft storage, and option generation.
- Reuse existing choice and branch semantics where practical, but centralize the durable node creation behavior.

## Non-Goals

- Do not add a standalone version history UI outside the tree.
- Do not introduce a second kind of persisted draft object.
- Do not allow editing finished sessions.
- Do not require visual merge tooling beyond this pass's diff editor controls.
- Do not change the public meaning of normal AI-generated option cards.

## Four Entry Types

### AI Option

The user chooses an AI-generated direction. The new child node direction is that selected option. The draft is generated by AI.

### Direct Edit

The user edits the visible draft in the normal editor. The new child node direction is fixed as `自定义编辑`. The draft is exactly the user-submitted draft, then the app generates the next three options from that draft.

### Diff Edit

The user enters diff mode, edits the compared draft, optionally reverts individual diff changes, and saves. The new child node direction is fixed as `自定义编辑`. The draft is exactly the diff editor's submitted draft, then the app generates the next three options from that draft.

### Custom Direction

The user writes a custom direction. The new child node direction is the user's custom option. The draft is generated by AI from that direction.

## Unified Flow

All entry types should call one durable creation path with these phases:

1. Resolve the parent node and selected direction.
2. Insert a new child node with parent linkage and the direction metadata.
3. Update the session's current node to the new child.
4. Insert the draft version for the child node.
5. Generate and persist the child's next three options.
6. Return fresh `SessionState`.

For AI-generated drafts, phase 4 receives the AI output. For user-submitted drafts, phase 4 receives the submitted draft and phase 5 uses option-only generation.

This order matters because the app should always be able to show that a node exists before its draft/options are filled, and every path should have the same state transition shape.

## Direction Model

Use a standard custom-edit option for direct and diff edits:

- `id`: `d`
- `label`: `自定义编辑`
- `description`: `根据用户手动编辑后的草稿继续。`
- `impact`: `保留这次手动修改，并从修改后的版本生成新的下一步方向。`
- `kind`: `reframe`

For normal custom directions, keep using the user-provided option content. For AI options, keep the selected generated option.

The parent node should record the selected direction in the same way for all paths, so folded options, branch history, and route selection remain coherent.

## API Design

Keep existing route surfaces where possible, but route them into shared internals:

- `POST /api/sessions/:sessionId/choose`
  - Handles AI options and custom directions that need AI draft generation.
  - Creates the child node first, then fills it with the generated draft and options.

- `POST /api/sessions/:sessionId/draft`
  - Changes from "edit current node in place" to "create custom-edit child node".
  - Accepts `{ nodeId, draft }`.
  - Creates the child node under `nodeId`, fills it with the submitted draft, then generates three options.

Diff editing can reuse the draft route because its result is just a submitted draft.

Historical branch generation should continue to work, but its implementation should also call the shared creation path when it creates a new child.

## Repository Design

Introduce a shared repository/service helper for draft node creation. The exact names can follow local style, but the responsibilities should be separated:

- Create a pending child node from a parent and selected direction.
- Activate that child node in the session.
- Save a draft version for the child.
- Save generated options on the child.
- Return state after each durable workflow completes.

The current `appendChoiceAndRound`, `createHistoricalBranch`, and `updateCurrentNodeDraftAndOptions` behavior should be refactored around this helper. `updateCurrentNodeDraftAndOptions` should stop mutating the active node and become a compatibility wrapper or be replaced by a user-draft child creation method.

## UI Design

### Normal Editing

The existing `编辑` button remains available for editable drafts. Saving changes its meaning:

- Old: `保存并重生成选项`
- New: `保存为自定义编辑`

After save, the tree focuses the new child node.

### Diff Editing

Diff mode should add an editing affordance when a full comparison draft is available. The editor starts from the displayed target draft. Users can make normal text edits and use quick controls on diff chunks to keep or revert changes where feasible.

Saving uses the same user-draft child creation path as normal editing. Canceling leaves tree state unchanged.

### Status And Busy States

While saving a user draft or generating options, keep using the existing busy state. If the node has been created but options are still generating, the UI can show the node with a loading/pending state using existing patterns.

## Data Flow

`TreeableApp` should pass enough context to `LiveDraft` for both normal edit and diff edit saves:

- Current viewed node id.
- Displayed draft.
- Comparison `from` and `to` drafts when diff mode is active.
- A single save callback that receives the submitted draft and the source mode.

`TreeableApp` should call the draft route with the currently viewed node id, not necessarily only `sessionState.currentNode.id`, because direct editing a viewed historical node should also create a child from that viewed node if editing is enabled for that context.

In the initial pass, editing can remain limited to the current node if that matches existing UI constraints, but the backend should not assume in-place mutation.

## Error Handling

- Return `404` if the parent node or session does not exist.
- Return `409` if the session is finished.
- Return `400` for invalid submitted draft payloads.
- If option generation fails after saving a user draft, keep the new child node and saved draft, return a clear error, and let the existing missing-options path retry generation when the node is viewed.
- If AI draft generation fails before a draft can be saved, keep the new child node only if the UI can clearly represent a pending or failed draft state; otherwise roll back that child node creation inside the request.
- Preserve API key friendly error messages in the client.

## Testing

Repository tests should prove:

- Direct edit creates a child node instead of updating the parent draft.
- The child node uses the `自定义编辑` direction and stores the submitted draft.
- The session current node changes to the child.
- The child receives three generated options.
- AI option, custom direction, direct edit, and historical branch creation all use the same child-node persistence behavior.
- Finished sessions and missing parents are rejected.

API tests should prove:

- `POST /draft` accepts a submitted draft and returns state focused on the new child.
- `POST /draft` no longer accepts stale or invalid parent nodes.
- `POST /choose` keeps normal AI and custom direction behavior while using the unified child node flow.

UI tests should prove:

- Normal edit save calls the user-draft save path and focuses the returned node.
- Diff edit initializes from the displayed target draft.
- Diff edit save submits the edited draft through the same callback as normal edit.
- Canceling diff edit leaves comparison and session state unchanged.

## Rollout Notes

This change affects existing assumptions in tests that editing mutates the current node. Update those tests to assert child-node creation. Keep route names stable so the frontend migration is small, but make the repository behavior the single source of truth.
